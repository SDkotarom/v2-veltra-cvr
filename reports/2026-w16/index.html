03Action — やるべきこと
課題は「計測基盤」— ユーザー体験施策ではなく bot 対応と KPI 二本化
見かけCVR悪化に引きずられて UI/UX 施策を振るのは誤り。bot除去で計測をクリーンにし、意思決定を正すのが最優先。
1
NEW / 今週中 WAF / インフラ 優先度: 最高
AWS ap-southeast-1 IPレンジ と fingerprint をWAFでブロック
今回のbotは 住宅IPプロキシを使っていない（AWSから直接発信）ため、IPレンジブロックだけで 88% 即除去できる。 AWS公開IPレンジJSONから ap-southeast-1 / EC2 を抽出して WAFルールに投入するだけ。半日工数で終わる。 補完として「Singapore × Chrome Win10 × Referer空 × /jp/japan/* アクセス」のfingerprintルールを追加し、残り12%もカバー。
ステップ: ① AWS IPレンジJSON取得（10分）② AWS WAF IPSet 作成 + Block Rule 追加（2時間）③ fingerprint補完ルール（4時間）④ CloudWatch で BlockedRequests 監視
2
NEW / 計測基盤 ダッシュボード 優先度: 最高
CVR を「raw」「bot除外」の2本併記に — 意思決定を誤らないため
今週の誤解の根本原因は raw CVR 一本で判断していたこと。GA4 探索レポートで Direct × Desktop を除外した参考CVRを並行計測し、 KPIダッシュボード / 週次サマリー / ボトルネックレポートの全てに2本併記する。 戦略ガイド「非bot比率90%以上」ルールを運用規則に昇格。
出力先: kpi.html / weekly-summary.json に cvr_raw と cvr_clean の2フィールド追加
3
中期 / 監視 アラート 優先度: 高
bot再来の早期検知アラート（次回候補 5/14前後）
月次バッチの可能性が高く5/14 前後の再来リスク大。 「Singapore × Direct × Desktop の日次セッションが 5,000 超」の条件で Slack #veltra-cvr-alerts に即通知する自動ルールを設定。 異常検知の基準は「Sessions WoW +20% 超」の週すべてで必ずデータ品質チェックを入れる運用に昇格。
4
継続調査 Organic × Mobile 優先度: 中
Organic Search × Mobile の月次 -20% 減速を並行調査
bot騒ぎと別に、最大セグメント（全体の約25%）の Organic × Mobile が月次 -20% 減速中。 月20万セッション・約1,400予約の逸失相当。Search Console で表示回数・順位・CTR の推移を確認し、SEO本体の問題か WAF副作用か切り分ける。
