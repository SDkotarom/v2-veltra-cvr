# Project 2: ペルソナ別カスタマージャーニーマップ（定量×定性ハイブリッド）

## ステータス: ブレスト段階

---

## 1. 概要

ペルソナごとの行動ジャーニーを、GA4等の実データと行動仮説を組み合わせて可視化するレポートページを作る。

週次のボトルネック分析が「今週どこが落ちたか」のスナップショットなのに対し、こちらは「このタイプのユーザーはそもそもどう動いているか」の構造的な理解を提供する。

**更新頻度**: 四半期に1回（季節要因の変化を捉える: GW / 夏休み / 秋シルバーウィーク / 冬年末年始）

## 2. ゴール

各ペルソナが「どこから入ってきて、どう動いて、どこで詰まるか」を1枚のジャーニーマップとして可視化し、施策の優先順位判断に使う。

**アウトプットイメージ（1ペルソナ分）:**

```
ペルソナA「初めての旅行者」

流入 ──→ 回遊 ──→ 検討 ──→ 意向 ──→ 完了
 │         │        │        │        │
 Organic   Area→    Ac到達   Calendar  Form
 Search    Ctg→Ac   42%      表示18%   完了45%
 68%                ↓                  
                    ここで落ちている
                    
推定行動: 一覧で比較しきれず、複数のAcページを行き来している
仮説: 一覧ページの比較情報が不足 / 候補が多すぎて絞れない
季節変動: GW前に流入増・CVR低下（検討期間が長い）
```

## 3. ペルソナ定義

### 3-1. 初期定義（Phase 0-1で使用）

GA4で取得可能なデータのみで定義する。

| ペルソナ | 定義（データ上の条件） | 推定される特徴 |
|----------|----------------------|---------------|
| A. 初めての旅行者 | 新規セッション × 予約未経験 | ベルトラを知らない。比較検討中。離脱しやすい |
| B. 検討リピーター | リピートセッション × 予約0-1回 | 何度か見に来ているが習慣化していない。迷っている |
| C. コアリピーター | リピートセッション × 予約2回以上 | 使い慣れている。ファネル通過率が高いはず |

**Phase 0での制約**:
- 「新規/リピーター」はGA4の `newVsReturning` ディメンションで取得可能
- 「予約回数」はGA4単体では取れない → Phase 0では新規/リピーターの2分割で開始
- 予約回数を加えるにはBigQueryでGA4エクスポート × 予約DBの結合が必要（Phase 2）

### 3-2. 拡張定義（Phase 2以降）

データソースが増えるにつれて、ペルソナの解像度を上げる。

| 追加データ | 可能になること | 必要なもの |
|-----------|---------------|-----------|
| 予約DB × GA4（BigQuery） | 予約回数ベースのペルソナ分割 | BigQueryエクスポート設定 + JOIN |
| User-ID（ログインユーザー） | クロスデバイストラッキング | GA4 User-ID設定 |
| Braze クリックデータ | メルマガ/LINE経由の復帰行動 | Brazeイベントの統合 |
| GTM スクロールイベント | ページ内のどこまで見たか | GTMタグ追加実装 |
| 在庫表示イベント | 在庫なしによる離脱の検知 | カスタムイベント実装 |
| フォームステップイベント | フォーム内の離脱ポイント | フォーム計測の追加実装 |

## 4. ジャーニーマップの構成要素

各ペルソナのジャーニーを以下の要素で記述する。

### 4-1. 定量レイヤー（実データ）

| 要素 | データソース | Phase |
|------|-------------|-------|
| 流入チャネル構成比 | GA4: sessionSource/medium × newVsReturning | 0 |
| ファネル通過率 | GA4: イベントベースファネル × newVsReturning | 0 |
| 代表的なページ遷移パス | GA4: パス分析（探索レポート） | 1 |
| セッション内ページ閲覧数 | GA4: screenPageViews / sessions | 0 |
| 離脱ページ Top5 | GA4: exits by pagePath × newVsReturning | 1 |
| クロスデバイス行動 | GA4: User-ID × deviceCategory | 2 |
| メルマガ→再訪→予約 | Braze + GA4 | 3 |

### 4-2. 定性レイヤー（仮説）

| 要素 | 生成方法 | Phase |
|------|---------|-------|
| 各段階での推定行動 | 定量データ + ページ構造から推定 | 0 |
| 離脱理由の仮説 | ファネル落差 + 2025年パターンから推定 | 0 |
| ペインポイント | 行動仮説の蓄積から抽出（Project 1連携） | 1 |
| 季節による行動変化 | 四半期比較（GW/夏/秋/冬） | 1 |

### 4-3. アクションレイヤー（施策示唆）

| 要素 | 内容 |
|------|------|
| このペルソナ向けの優先改善ポイント | ファネルで最も落差が大きい段階 |
| ページ役割チェック | 落差のある段階のページは役割通りに機能しているか |
| 推奨施策方向 | 引き算/視認性向上/並走型（2025年勝ちパターン準拠） |

## 5. 必要なデータの一覧（優先順位付き）

### すぐ取れる（GA4標準 × MCP）

| データ | GA4ディメンション/メトリクス | 用途 |
|--------|---------------------------|------|
| 新規/リピーター別セッション数 | newVsReturning × sessions | ペルソナ分母 |
| 新規/リピーター別ファネル | newVsReturning × eventName × eventCount | ペルソナ別通過率 |
| 新規/リピーター別チャネル | newVsReturning × sessionDefaultChannelGroup | 流入経路の違い |
| 新規/リピーター別デバイス | newVsReturning × deviceCategory | デバイス利用傾向 |
| 平均ページビュー数 | screenPageViews / sessions（セグメント別） | 回遊の深さ |

### 取れるが追加実装が必要

| データ | 必要な実装 | 優先度 | 用途 |
|--------|-----------|--------|------|
| ページ遷移パス | GA4探索レポート（手動 or BigQuery） | 高 | 代表的な行動パス |
| 離脱ページ分析 | GA4: exitPage（MCP対応要確認） | 高 | どこで止まるか |
| スクロール深度 | GTM: scroll イベント追加 | 中 | Acページ内の到達度 |
| セクション表示 | GTM: element_visible イベント | 中 | どの情報を見たか |
| 在庫なし表示 | カスタムイベント実装 | 中 | 検討段階の離脱理由 |

### 取れないが将来欲しい

| データ | 必要な基盤 | 用途 |
|--------|-----------|------|
| 予約回数別セグメント | BigQuery（GA4 + 予約DB JOIN） | ペルソナ精緻化 |
| クロスデバイス行動 | GA4 User-ID + ログイン促進 | SP調べ→PC予約パターン |
| メルマガ→再訪→予約 | Braze + GA4 連携 | リピーター育成評価 |
| カート放棄→復帰率 | Braze リマインド + GA4 | 離脱→復帰の全体像 |

## 6. 段階的ロードマップ

### Phase 0（今すぐ — Project 1と同時着手）

**やること**: 新規/リピーター × ファネル通過率のクロス表をdata.jsonに追加

これはProject 1の「ボトルネック分析の解像度UP」と完全に同じ作業。新規/リピーターのセグメント軸を既存レポートに足す。ジャーニーマップの定量骨格でもある。

**成果物**: 週次レポートのFunnel Analysisに新規/リピーター行が追加される

### Phase 1（Q2 — ジャーニーマップv1）

**やること**:
- GA4探索レポート（パス分析）でペルソナ別の代表的ページ遷移を抽出
- 定量データに行動仮説を重ねて、ジャーニーマップv1を作成
- 離脱ページ Top5 を取得してペインポイントを特定
- Project 1で蓄積された行動仮説を統合

**成果物**: レポートサイトに `/journey/` ページを追加（四半期更新の固定ページ）

**GAP**: GA4パス分析がMCP経由で取れるか未検証。手動抽出→JSONの可能性あり

### Phase 2（Q3 — データソース拡張）

**やること**:
- BigQuery連携で予約DB × GA4を結合 → 予約回数ベースのペルソナ定義
- User-IDベースのクロスデバイス分析（ログインユーザー限定）
- GTMでスクロール深度・セクション表示イベントを追加

**成果物**: ジャーニーマップv2（ペルソナ3分割、クロスデバイス行動あり）

### Phase 3（Q4〜 — 行動ログ統合）

**やること**:
- Braze連携（メルマガ/LINE/プッシュ→再訪→予約のトラッキング）
- 在庫なし表示、フォームステップの計測追加
- 季節比較（Q2 vs Q3 vs Q4）による行動変化の分析

**成果物**: ジャーニーマップv3（季節変動あり、タッチポイント横断）

## 7. 季節分析の設計（四半期更新時）

| 四半期 | 期間 | 日本人アウトバウンドの特徴 | 分析の焦点 |
|--------|------|--------------------------|-----------|
| Q1（1-3月） | 冬〜春 | 年末年始明け。卒業旅行シーズン | 新規流入の質（学生セグメント？） |
| Q2（4-6月） | 春〜初夏 | GW。ファミリー・カップル中心 | GW前の検討期間の行動パターン |
| Q3（7-9月） | 夏 | 夏休み。最大繁忙期 | 流入増に対するCVR維持 |
| Q4（10-12月） | 秋〜冬 | シルバーウィーク→年末年始準備 | リピーター比率の変化 |

四半期ごとにジャーニーマップを更新し、前四半期との差分を出す。「このペルソナは夏に〇〇な行動をとるが、冬は△△に変わる」を記録する。

## 8. Project 1 との関係

```
Project 1（週次）                    Project 2（四半期）
┌─────────────────┐                ┌──────────────────────┐
│ ボトルネック分析  │                │ ジャーニーマップ       │
│ + 行動仮説       │───蓄積───→    │ ペルソナ別行動構造    │
│                  │                │ + 季節変動            │
└─────────────────┘                └──────────────────────┘
     ↑                                      │
     │              施策の優先順位           │
     └──────────────フィードバック───────────┘
```

- Project 1で毎週蓄積される行動仮説が、Project 2のジャーニーマップの素材になる
- Project 2のジャーニーマップが、Project 1のボトルネック分析の解釈を深くする
- 両方のPhase 0は同じ作業（新規/リピーター軸の追加）
