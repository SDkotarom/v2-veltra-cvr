### やらない（スコープ外・EOL まで手を出さない）

- AC ページ ⇄ booking URL の統合／URL 設計変更
- AC ⇄ booking またぎの戻り（token 保持・履歴連続性）
- 画面構成そのものの根本改修（5画面→3画面統合など）
- 入力UI ⇔ 確認UI の見た目刷新
- ACページ側のステップガイド（元施策 BN1-F）— プリフィル動作で代替可能なため
- モバイル既存タブの即時撤去 — 計測先行・A/Bテストで判断

### 開発に確認（P1 投入可否を判断）

- **2→3 ローディング短縮**: バックエンド処理で削れる箇所があるか。
  視覚差し替え（スピナー→スケルトン）だけでは効果薄。実時間短縮ができるなら P1、不可なら P2 以降。

## 4. 矢羽仕様

### ステップ定義（デバイス別）

| | モバイル | PC |
|---|---|---|
| ステップ数 | **4** | **3** |
| ① | 日程 | 予約詳細 |
| ② | 参加者情報 | お客様情報 |
| ③ | 支払方法 | 支払方法 |
| ④ | 内容確認 | —（サイドバーで代替） |

### 画面 ⇔ ステップマッピング（モバイル）

| 画面 | active | done |
|---|---|---|
| 画面1 (カレンダー) | ① | — |
| 画面2 (利用数・時間) | ① | — |
| 画面3 (お客様情報) | ② | ① |
| 画面4 (支払・ポリシー) | ③ | ①② |
| 画面5 (内容確認) | ④ | ①②③ |

### UI要件

- **全ステップを常時表示**（完了・現在・残り全部見える）
- **残り◯ステップ** テキストを矢羽横 or 下に添付（「あと2ステップ」）
- 状態カラー:
  - active: `#1B82C5`
  - done: `#2DAE6C` + ✓
  - upcoming: `#AAAAAA`
- インタラクション:
  - done = タップで戻れる（**下線で明示**／state 保持）
  - upcoming = 無効・操作不可
  - ステップ順固定（進行方向スキップ不可）
- 配置: 各画面の**最上部 sticky**（PC/SP 共通）
- モバイル既存タブは P1 では**矢羽と並存**（撤去判断は計測データ取得後）

---

## 5. フォームUX原則（設計の指針）

- **一方通行・最小限の操作で進める**
- 離脱リンクを極力削除（決済直前のナーバスなユーザーを迷わせない）
- ミス修正のための戻りだけ残す（done ステップタップに集約）
- 余計な CTA・選択肢を出さない

## 6. 計測プラン

### 新規イベント（GTM で実装）

| イベント名 | トリガー | パラメータ |
|---|---|---|
| `booking_step_view` | 各ステップ画面表示時 | `step` (1-4), `device` (sp/pc), `prefilled` (true/false) |
| `booking_step_complete` | 「次へ」押下成功時 | `step`, `device` |
| `booking_step_back` | done ステップタップで戻った時 | `from_step`, `to_step` |

### 追加計測（タブ撤去判断用）

| イベント名 | 対象 |
|---|---|
| `booking_tab_click` | モバイル既存タブ（予約詳細／お客様情報／お支払方法） |

### A/B テスト設計（P2以降）

| | A | B |
|---|---|---|
| 矢羽 | あり | あり |
| モバイルタブ | 現行維持 | 撤去 |
| 判定指標 | `booking_step_complete` 到達率（step=4） |

## 7. 開発チケットドラフト

**施策名**: 予約フローにステップ矢羽を追加 / Add Step Progress Indicator to Booking Flow

**対象**: 予約フロー全画面（SP=5画面・PC=3画面）、JP 日本語サイト

**機能**:
- デバイス別矢羽（SP=4ステップ・PC=3ステップ）
- 全ステップ常時表示＋残◯ステップテキスト
- 状態管理（active/done/upcoming）
- done タップで戻り（データ保持）
- SP に「前に戻る」ボタン追加
- SP の内容確認画面から各ステップへ直接遷移
- GA4 計測イベント 4種（step_view / _complete / _back / tab_click）
- **A/Bテスト前提で実装**（タブあり/なしの切り替え可能に）

**受け入れ要件**:
1. 全画面で矢羽描画（SP/PC 各デバイス別）
2. 画面⇔ステップマッピング通りの強調
3. done タップで戻り（データ保持）、upcoming タップ無効
4. 残り◯ステップ表記
5. SP で「前に戻る」ボタンが各画面に配置
6. SP 内容確認画面から各ステップへ戻るリンク
7. 矢羽・タブ共存（P1）／タブ切替可能な設計
8. `booking_step_*` イベント発火確認

**開発への確認事項**:
- 画面2→3 の処理時間実測と短縮可否（実時間短縮できれば P1 に含める）

**フェーズ**:
- P1: 矢羽 + SP戻る + 確認画面からの戻り + 計測
- P2: タブ撤去判断（A/Bテスト結果で）
- P3以降: AC側対応・URL統合は**やらない**（EOL 方針）
