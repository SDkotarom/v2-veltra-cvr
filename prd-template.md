## 2. エグゼクティブサマリー

**目的:** プロジェクトの概要・目標・期待成果を、ステークホルダーが素早く理解できるよう簡潔にまとめる。

**記入例:**
> 本プロジェクトでは、既存Webアプリケーションに統合する「顧客フィードバック＆アイデアポータル」の開発を計画している。ユーザーがアイデアの投稿・投票・コメントを行える場を提供し、プロダクトチームに構造化されたインサイトを届けることで、顧客満足度とプロダクトのイノベーションを向上させる。

---

## 3. 課題 / 機会の定義

**目的:** 解決すべきユーザー課題、または獲得すべき市場機会を明確にする。どんなペインポイントがあるか？ どんなギャップを埋めるか？

### 課題
[ユーザーが直面している具体的な問題を記述]

**記入例:**
> 顧客がプロダクトへのフィードバックや機能要望を提出する一元的な手段がない。フィードバックはサポートチケット、メール、SNS、個別の会話に散在しており、プロダクトチームが追跡・優先順位付け・対応を効果的に行うことが困難。

### 機会
[課題を解決することで得られるビジネス上の機会を記述]

**記入例:**
> 専用ポータルを提供することで、ユーザーから直接、価値のあるアクショナブルなインサイトを取得し、コミュニティ意識を醸成し、ユーザー主導の開発へのコミットメントを公に示すことができる。

## 4. ゴール＆目標（SMART Goals）

**目的:** プロジェクトの成功を定義する。具体的（Specific）・測定可能（Measurable）・達成可能（Achievable）・関連性のある（Relevant）・期限付き（Time-bound）であること。

**記入例:**
- ローンチ後3ヶ月以内に、構造化されたフィードバック投稿数を50%増加させる
- ローンチ後6ヶ月以内に、機能要望に関するサポートチケットを25%削減する
- ローンチ後4ヶ月以内に、ポータル体験の平均CSAT（顧客満足度）スコア4.5/5を達成する
- 12ヶ月以内に、上位投票アイデアの75%をプロダクトロードマップに統合する

## 6. ソリューション概要 / ハイレベルデザイン

**目的:** 提案するソリューションをハイレベルで記述する。何を作るか？ どう動くか？ コアなユーザー体験は？

**記入例:**
> Webアプリのメインナビゲーションからアクセスできる新セクション。ユーザーは既存アイデアの閲覧、新規アイデアの投稿、投票、コメントが可能。各アイデアにはプロダクトチームが更新するステータス（例：「レビュー中」「計画済」「対応中」「完了」）が表示される。検索・フィルター機能を搭載。社内チーム向けにはプライベートな管理画面を提供。

---

## 7. 主要機能

**目的:** ソリューションが提供する具体的な機能を一覧化する。エピックまたはハイレベルなユーザーストーリーで整理する。

### ユーザー向け機能
- [ ] **アイデア投稿**: タイトル、説明、カテゴリ、添付ファイル付きフォーム
- [ ] **アイデア閲覧・検索**: 全アイデア一覧 + 検索バー + フィルター（カテゴリ、ステータス、人気順）
- [ ] **投票**: アイデアへの賛成・反対投票で優先度を表現
- [ ] **コメント**: アイデアへのコメント・ディスカッション
- [ ] **ステータス表示**: 各アイデアの進捗ステータスを可視化
- [ ] **ユーザープロフィール**: 自分の投稿・投票履歴の確認

### 管理者向け機能
- [ ] **アイデア管理**: ダッシュボードでの閲覧・編集・分類・アーカイブ
- [ ] **ステータス管理**: アイデアの公開ステータス更新
- [ ] **モデレーション**: コメント・投稿の管理
- [ ] **レポーティング**: 投稿量・人気アイデアの基本分析
- [ ] **外部連携**: Jira等との双方向同期

---

## 8. 非機能要件

**目的:** システムの動作特性を定義する（パフォーマンス、セキュリティ、ユーザビリティ、スケーラビリティなど）。

| カテゴリ | 要件 |
|---|---|
| パフォーマンス | 1,000件のアイデア一覧表示で2秒以内のページロード |
| セキュリティ | OWASP Top 10準拠、GDPR対応、ユーザーデータ保護 |
| スケーラビリティ | アクティブアイデア10,000件、月間ユニークビジター100,000人に対応 |
| ユーザビリティ | 非技術ユーザーにとって直感的なUI/UX |
| 互換性 | デスクトップ・タブレット・モバイルのレスポンシブ対応 |
| 認証連携 | 既存認証システムとのシームレスな統合 |

---

## 9. 成功指標・KPI

**目的:** ゴール達成をどう測定するか（セクション4と連動）。

- 月間新規アイデア投稿数
- 月間ユニークインタラクションユーザー数
- アイデアあたり平均投票数
- アイデア投稿からステータス更新までの平均時間（社内指標）
- 人気アイデアのロードマップ反映率
- ポータル機能のCSATスコア

---

## 10. スコープ外

**目的:** プロジェクトに含まれない内容を明示する。期待値管理とスコープクリープ防止に重要。

**記入例:**
- ユーザー間のアプリ内ダイレクトメッセージ
- 汎用コミュニティフォーラム機能（アイデアに紐づかない一般的なディスカッション）
- サードパーティからのフィードバック投稿用パブリックAPI
- 外部アンケートツールとの直接連携（初期リリースでは対象外）
- ゲーミフィケーション要素（ポイント、バッジなど。基本的な投票を除く）

## 11. 技術的考慮事項（ハイレベル）

**目的:** 主要な技術アプローチ、アーキテクチャ判断、インフラ要件の概要。

| 領域 | 方針 |
|---|---|
| フロントエンド | 既存アプリフレームワーク内で [React/Vue等] を使用 |

| 認証 | 既存 OAuth2/SSO システムを活用 |
| API | フィードバックの投稿・取得・管理用 RESTful API を新規開発 |

## 13. 依存関係＆リスク

### 依存関係
- デザインリソースの確保
- バックエンド認証更新の完了
- [特定エンジニアリングチーム] のAPI連携キャパシティ

### リスク
| リスク | 影響度 | 対策案 |
|---|---|---|
| 新ポータルのユーザー定着率が低い | 高 | ローンチ前にベータテストで検証、段階的な導入 |
| スパム・低品質投稿の大量発生 | 中 | モデレーションキューの導入 |
| 既存ツールとの連携の複雑さ | 中 | 早期のPoCで技術検証 |
| ステークホルダー要望によるスコープクリープ | 高 | スコープ外を明示、変更管理プロセスの徹底 |
| フィードバックへの対応が遅い場合の顧客不満 | 高 | SLA設定、ステータス更新の自動通知 |

## 使い方

1. この構造をコピーする
2. `[...]` と記入例をプロジェクトの実際の内容に置き換える
3. 各セクションで以下を検討する：
   - **どんな課題を解決するか？** → 課題/機会の定義
   - **誰のために？** → ターゲットユーザー
   - **成功とは？** → ゴール＆目標、成功指標
   - **何を作るか？** → ソリューション概要、主要機能
   - **何を作らないか？** → スコープ外
   - **品質の制約は？** → 非機能要件
   - **大まかな計画は？** → タイムライン、技術的考慮事項、依存関係
   - **誰が関わるか？** → ステークホルダー
   - **リスクは？** → 依存関係＆リスク
   - **未確定事項は？** → 未解決事項

> コンテキストが詳細で明確であるほど、PRDの品質と精度が向上します。
